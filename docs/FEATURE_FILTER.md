# Propelo Feature Filter

## 1. Purpose
Propelo exists to build an AI-guided system architecture workspace, not a generic productivity app. This filter protects the product from feature creep by forcing every proposed change to prove it strengthens the architectural thinking model instead of adding surface-level conveniences.

## 2. Golden Question
**Does this improve architectural thinking?**
If the answer is not an immediate yes, the feature pauses here.

## 3. The Propelo Pipeline
Every capability must reinforce the core flow:
- **Mission** – capture intent and constraints.
- **Decisions** – lock guardrails, assumptions, and risks.
- **Blueprint** – design the phased execution (phases, boundaries, automation map).
- **Tasks** – turn blueprint nodes into proof-of-work execution.
- **Project Memory** – capture wrap snapshots so the system can resume with context.

## 4. Allowed Feature Types
Only features that deepen system architecture thinking make the cut:
- **Decision intelligence** (prompts, validation, impact analysis).
- **Risk mapping** (linking risks to decisions, phases, and tasks).
- **Blueprint visualization** (phase flows, dependencies, automation maps).
- **Task generation** (phased, proof-of-work ready tasks driven by blueprint logic).
- **Architecture snapshots** (wrap packets, project memory exports, readiness indexes).

## 5. Disallowed Feature Types
The workspace may never drift into generic tooling. Reject features that resemble:
- Generic notes or sticky pads.
- Flexible/blank documents.
- Chat systems without architectural context.
- Arbitrary pages or page builders.
- Wiki/knowledge-base systems.

## 6. AI Feature Rule
AI belongs only where it strengthens reasoning and structure. Valid uses include:
- Asking clarifying questions tied to mission/decisions.
- Suggesting decisions or assumptions with explicit confidence.
- Detecting risks or scope conflicts.
- Proposing blueprint adjustments or phased tasks.
AI may never introduce free-form chat, unstructured knowledge dumps, or automated execution without the Propelo pipeline.

## 7. The Structure Rule
Propelo must never become a blank workspace. Users interact exclusively with structured objects:
`mission`, `decision`, `risk`, `phase`, `task`, `snapshot`.
If a proposed feature introduces a new object, it must prove the pipeline needs it.

## 8. Final Principle
**Propelo is a thinking operating system for designing systems.**
